## やりとりの流れ

### 1. スタッフレビュー指摘の検討

**スタッフ指摘（2件）:**
- No.1: ALB internal化の前にIP制限で暫定対応する2段階アプローチを提案
- No.2: 顧客がSTG Web UIを使用するか事前確認が必要。使用する場合NAT/DNS設定が別途必要

**きよはらさんの指摘:**
- スタッフの意見を鵜呑みにするな。間違っている可能性を考えろ
- カスミ様は既にGIFT端末等で閉域網経由アクセスしている実績がある
- STGで動作検証しないで本番リリースはありえない

**結論:**
- No.1: **不採用**（閉域網基盤が既に稼働、暫定IP制限は不要）
- No.2: **指摘の前提が誤り**（STG検証は当然実施、閉域網アクセス実績あり）

### 4. DNS考慮事項の追加

きよはらさんから以下の指摘:
- 過去にカスミ様から「AWS内のIPは変動するのでドメインで指定すべき」との意見あり
- 実際にIP変動による障害が発生した実績がある
- 顧客は `https://stg.ignicapos.com/login` でアクセスできるのか？

**調査結果（Route 53 / PRDアカウント）:**
```
stg.ignicapos.com      → CNAME → ksm-posstg-alb-web-fe-541431683...elb.amazonaws.com
api-stg.ignicapos.com  → CNAME → ksm-posstg-alb-api-be-1968385482...elb.amazonaws.com
web-stg.ignicapos.com  → CNAME → ksm-posstg-alb-web-fe-541431683...elb.amazonaws.com
```

**結論:**
- ALB internal化後もALB DNS名は不変（IPのみprivateに変動）→ CNAME変更不要
- `stg.ignicapos.com` でアクセス可能（DNS解決チェーンが機能すれば）
- テストC-1をIP直指定 → ドメイン名ベースのnslookup確認に修正
- IP直指定禁止の原則を明記（過去障害の教訓）

### 5. Route 53 環境分離の課題

きよはらさんの指摘:
- PRDとSTGでRoute 53を共有しているのは正しいのか？

**回答:**
- AWSベストプラクティスに反する。環境分離しているならRoute 53も分けるべき
- サブドメイン委任（delegation）が推奨

→ **KSM-AWS-027 として別途起票**
