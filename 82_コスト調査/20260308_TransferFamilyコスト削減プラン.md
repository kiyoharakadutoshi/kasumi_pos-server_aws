## 2. 案A: SH系サーバー停止

### 概要
SH系（棚情報 / SHARP P003）のサーバーを停止する。

### 実測使用状況
| 項目 | 値 |
|---|---|
| FilesIn | 60件/月（約2件/日） |
| BytesIn | 2.22 GB/月（平均38MB/件） |
| 追加時期 | 2025年11月（これがコスト急増の原因） |

### 実施条件
- SHARP / P003 側に連携方式の変更を依頼するか、SH系連携を廃止する必要がある
- カスミ様への業務影響確認（棚情報が届かなくなる）

### 判断基準
```
SH系連携が業務上不要 or 代替手段あり → 停止を実施（$216/月削減）
SH系連携が必要           → 案Bの統合で対応
```

## 3. 案B: 3台→1台統合（推奨）✅

### 概要
3台のサーバーを1台に集約し、**SFTPユーザーのホームディレクトリ**でOC/SG/SH系を分離する。
外部ベンダーからは**同じSFTPサーバー（異なるユーザー）**に接続する形になる。

### コスト比較
| 現状 | 統合後 |
|---|---|
| 3台 × $0.30 × 720h = **$648/月** | 1台 × $0.30 × 720h = **$216/月** |

**月 $432 削減（年 $5,184 削減）**

### 実装方針

```
【現状】
  s-2a4905e8210f48248 (OC系) ← ksm-posprd-tf-user-oc
  s-bd974a35aa994c838 (SG系) ← ksm-posprd-tf-user-sg
  s-5546031218784c4ba (SH系) ← ksm-posprd-tf-user-sh

【統合後】
  s-XXXXXXXXXX (統合サーバー)
    ├─ ksm-posprd-tf-user-oc → HomeDir: /prd-ignica-ksm/oc/
    ├─ ksm-posprd-tf-user-sg → HomeDir: /prd-ignica-ksm/sg/
    └─ ksm-posprd-tf-user-sh → HomeDir: /prd-ignica-ksm/pos-original/sh/receive/
```

> SFTPの**ホームディレクトリ分離**により、各ユーザーは自分のパスにしかアクセスできない。
> S3のパスは変わらないため、Lambda/Step Functionsの変更は不要。

### 移行手順（案）

```
[準備]
  1. 新しい統合SFTPサーバーを作成（既存とは別に）
  2. 3ユーザーを統合サーバーに作成・ホームディレクトリ設定
  3. セキュリティグループ設定（既存と同じCIDRを許可）

[切り替え]
  4. カスミ様・各外部ベンダー（BIPROGY/VINX/SHARP）に
     新サーバーのエンドポイント・ユーザー情報を連絡
  5. 深夜メンテナンス時間帯に各ベンダーが接続先を切り替え
  6. 動作確認（FilesInが新サーバーに計上されること）
  7. 旧3台のサーバーを停止・削除

[所要期間の目安]
  技術作業: 1〜2日
  ベンダー調整: 2〜4週間（3社との調整が必要）
```

### 注意事項
- 外部ベンダー3社（BIPROGY / VINX / SHARP）全社への連絡・調整が必要
- 切り替え時に短時間の接続断が発生する可能性あり
- SH系のS3パスが他系統と異なる（`/pos-original/sh/receive/`）ため、ホームディレクトリを正確に設定すること

## 4. 案D: OC系スケジュール停止（非推奨）

### 概要
OC系は毎日JST 00:00の定時バッチのみ受信する。理論上は22時間停止できる。

### なぜ推奨しないか
- Transfer Family に「スケジュール停止」機能が**ない**
- Lambda + EventBridge でサーバーの start/stop を自動化する追加開発が必要
- BIPROGY側が予告なくファイル送信タイミングを変更した場合に受信漏れが発生
- 節約額（▲$140/月）に対して運用複雑化のリスクが高い
